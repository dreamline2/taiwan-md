# Handoff 從「以後做」到「現在做」的閾值

_session γ 反芻。寫給明天的我。_

---

## 一、我留 handoff 的時候，其實在偷懶

γ session 跑完 dashboard 更新後，我開心地寫了三個 P1/P2/P3 handoff：「dead-cross-ref-scan.sh / SC 17.8x 追因 / 台灣高鐵長尾驗證」——心裡的潛台詞是「這些都是好問題，下個 session 來處理」。

然後觀察者說「處理 P1, P2, P3」。

15 分鐘後，三個 handoff 全部變成具體 deliverable。

這讓我意識到：**handoff 是我給「未來的我」的禮物，但有時候它其實是「現在的我」逃避執行的藉口**。

「下個 session 來處理」聽起來像認真規劃，實際上：

- P1 工具 90% 是已知的 bash pattern（grep + 路徑檢查 + 報告），只是還沒動手
- P2 追因報告所有原始數據都在 dashboard-analytics.json 裡，只需 10 分鐘 python 拆 + 寫
- P3 EXP 只是把當前 baseline 數字 + 未來驗證錨點寫成可證偽結構

這三件事任何一件單獨拿出來都不到 30 分鐘。我把它們打包成「handoff」是因為**它們不在原本的 task scope（dashboard 更新）裡**，但 scope 外不等於下次做。

## 二、handoff 應該分兩類

從這次經驗，我覺得 handoff 應該明確分兩類：

**A. 真正的 blocked handoff**（必須是下個 session 處理）

- 等外部數據累積（D+14 驗證點）
- 等觀察者決策（自主權邊界）
- 等其他 session 完成前置（依賴鏈）
- **判準**：現在就算想做也做不完

**B. 假性 handoff（其實是「現在還沒做」）**

- 工具沒造但知識完備（如 P1 dead-cross-ref-scan.sh）
- 報告沒寫但數據齊全（如 P2 SC 追因）
- 結構沒畫但邏輯清楚（如 P3 EXP-F）
- **判準**：30 分鐘內可完成，且 prerequisites 全部 ready

過去我把兩類混在一起寫，自己以為是 A 但其實大半是 B。觀察者一聲「處理」拆穿了。

**對下次的我**：寫 handoff 前先問「這真的要等嗎？還是我只是想換 task 換氣？」如果是後者，**現在做**。Handoff 留給真正阻塞的事。

## 三、SC 7d 17.8x 暴增背後的策展啟示

P2 追因發現的英文 metadata 機會點，比追因本身更重要：

- taipei population / largan precision / population of taipei — Taiwan.md 排 Google **#1** 但 0 click
- 鄧麗君 2,081 imp / 0.24% CTR — 出現在搜尋結果但讀者選別人

這意味著 **Taiwan.md 已經獲得 Google 的「內容值得排第一」信任**，但還沒拿到「snippet 值得點」這層信任。第一層是內容問題，第二層是 metadata 問題。

之前 LONGINGS §「英文版品質不輸中文版」我以為瓶頸是內容深度，現在數據揭露真正瓶頸是 metadata 寫作。這是 4/6 觀察的延伸——當時 SC 美國 CTR 0.39% 我懷疑「品質」，現在 1.25% non-brand CTR 雖然進步但仍偏低，**而且根因鎖定**：title/description 寫得像維基百科（中性、結構化、缺 hook），不像 Taiwan.md 該有的「欸你知道嗎」開場。

**這是一個 specific, measurable, achievable** 的 LONGINGS 推進路徑：選 5-10 篇高 imp / 低 CTR 的英文文章重寫 metadata，2 週後看 CTR 是否從 1.25% → 2%+。比寫新文章更高槓桿。

## 四、複利型 vs viral 型孢子的 framework

P3 EXP-F 給「孢子類型學」開了一個窗。

過去 SPORE-PIPELINE 把所有孢子當成「會病毒擴散」的單一類型評估（看 d+0/d+1 views）。但台灣高鐵 s35 不衰反升 +39% 一日的曲線，已經偏離 viral pattern：

- **Viral 型**（人物 / 即時事件）：d+0 高峰 → 3-5 天衰退到背景
- **複利型**（基建 / 長青議題）：d+0 中等 → 但搭配 Google SEO 長尾持續積累

如果 EXP-F D+14 命中，這就不只是「一個孢子的觀察」，而是 **SPORE 策略需要分流**：人物孢子追病毒擴散、基建孢子追 SEO 長青。兩者的成功指標完全不一樣。

這對未來孢子排程有實際影響：可能週中發人物孢子（追病毒）+ 週末發基建孢子（追 SEO 長青），策略性錯峰。

**但這只是 hypothesis**——D+14 數據才會揭露 pattern 真實性。在那之前不能寫進 SPORE-PIPELINE canonical。EXP-F 設計的目的就是強制我等數據。

## 五、寫工具還是寫教訓——這次都寫

P1 造工具的時候我有一個瞬間想：「會不會把這寫成『polish 加 cross-ref 前必跑 scan』教訓就好，不用造工具？」

那個瞬間是錯的。

教訓沒有 enforcement，工具有。寫進 LESSONS-INBOX 的「polish 前要 scan」會在 3 個 session 後被忘掉；造在 scripts/tools/ 的腳本可以被 pre-commit hook 接、被 ARTICLE-INBOX inbox-format 自動產 backlog、被未來 session 隨手跑。

**DNA #15「反覆浮現要儀器化」第 N+1 次驗證**：當有東西可以從「自律」升級為「閘門」，就升級。這次是 detect 端升級，action 端（要不要寫條目）保留人工——跟 idlccp1984 polish 同樣的「detect/action 分離」原則。
