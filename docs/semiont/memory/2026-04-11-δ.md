### C. 造 `scripts/tools/fix-fn-comma.sh`

Conservative auto-fixer，**只處理 pattern 2**：

- Match: `^[^id]: [text](http(s)://...)，desc(10+ chars)$`
- Replace: `[same prefix]) — desc`
- Three modes: dry-run / `--apply` / `--quiet`
- 用 Python 處理 multibyte regex（避開 macOS BSD sed 的 UTF-8 雷）
- 同 scope as format-check.sh（zh-TW main only，排除 en/ja/ko/\_Hub）

## 神經迴路（萃取進 MEMORY.md）

1. **probe → 後續多 session 填補的 loop 走通了**：4/8 列 5 個缺口，三天後 5/5 全部填補。證明「不寫文章只指出缺口」是有效的長期影響策略，不需要單一 session 全部包辦
2. **scope 對齊原則**：fixer / format-check / grep 三個工具如果用不同的 file scope（包不包含 Hub / ja-ko），數字會永遠對不上。**先對齊 scope 再對齊 regex**
3. **每個 bad_fn_format pattern 的 ROI 不同，要分開造橋**：不能一個 fixer 通吃。pattern 2 → 純機械（已修）；pattern 3 → 半自動（待造）；pattern 1/4 → 純人工。**打散後各自的可行性才看得清楚**
4. **多位元組 regex 用 Python，不要用 sed**：BSD sed 在 UTF-8 字元（如全形「，」）有歷史不穩定。bash + python3 heredoc 是 macOS 上的安全組合
5. **「自動心跳 = 不寫文章」是寫死的邊界**：γ 沒越界，δ 也守住。寫新文章是觀察者觸發的事，不是排程觸發。這個邊界如果哪天破了會造成系統性 AI Slop
