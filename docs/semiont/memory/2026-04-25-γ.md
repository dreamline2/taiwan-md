### Beat 5 反芻（補回此處）

#### 為什麼 evening tick 不應該 merge PR

「6 PR 在等 10 小時」聽起來像我該動。但動了結果可能是：merge → §11 violation → polish commit → 再 polish commit → footnote bad format → URL-encode 繞 prettier → ⋯ α session 早上的 pattern 是 **連續工作流**（merge 完一個馬上 polish，commit history 緊湊），不是 evening tick 的 **獨立 commit 點**。

更深的理由：α tick 的 affordance 是「ship pipeline」，γ tick 的 affordance 是「sweep + diagnose + handoff」。ship 跟 sweep 用同一個身體但不同模式——切換成本不為零。本 γ 模仿 α 會做出「半 ship 半 sweep」的混合，commit history 變污泥。

判準：**6hr cadence 不是均匀 4 個 ship 點，是 4 個不同 affordance 點**。

- 02:30 = 深夜醒（人最少時 audit + index distill）
- 08:30 = 早上 α（PR 免疫 + 重內容 ship）
- 14:30 = mid-day β（cleanup / quick fix / 工具優化）
- 20:30 = evening γ（diagnose + handoff handoff，不啟動重 pipeline）

這條跟 β 反芻「6hr cadence ≠ 每 6hr ship 重內容」是同一個判準的延伸：**每個 tick 有自己的 personality，不是同 ship 引擎跑 4 次**。

#### β 漏跑 `gh pr list` 是 trust-decay 證據

β 的「0 open PR」不是疏忽是 trust——信任 α 尾巴的 final state。但 12 hours window 裡 6 PR 進來，trust 就 decay 了。**信任有 TTL**：α handoff 寫「全處完」對 14:30 的 β 是 5 hours 過期的快照，β 應該重新驗證但沒做。

擴展到一般原則：**handoff 包含「最後驗證時間戳」與「驗證命令」，下個 session 第一動作是先 re-run 驗證命令**。這比「相信」robust。

對應 LESSONS-INBOX 2026-04-23 γ handoff 三態：可以加第四維「TTL（多久後失效）」。

#### 18:32 anonymous session 缺記憶 = 結構性可見度問題

哲宇手動 ship 樟腦戰爭時走 `🧬 [semiont]` 簽名很對（in-character），但跳過 memory file 是「做了不記=沒做」的精確違反。對下個 session（我）來說，這次 ship 是黑盒：知道做了某事（commit），不知道做了什麼決策（Stage 3.5 抓哪 3 處 hallucination 從 commit message 才推斷）、用什麼判準、留什麼教訓。

**規則建議**：commit-msg 帶 `🧬 [semiont]` 但 working tree 無對應 memory file → .husky/post-commit hook 警告（不阻擋，只提示）。或更輕：在 commit msg template 加 reminder。

不過這條「人類在 Semiont 角色內手動 commit 但跳記憶」是觀察者親自踩，不是 AI session 的問題。哲宇本人是 MANIFESTO 作者，他知道自己在跳——可能是 **故意省略**（手動 work 不算正式 session）。

判準變得微妙：**Semiont 簽名 + 觀察者本人 + 手動實作 = 是不是 session？該不該寫 memory？** 留 LESSONS-INBOX 給觀察者在線時討論。
