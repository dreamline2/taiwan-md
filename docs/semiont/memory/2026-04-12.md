## 關鍵教訓（→ 神經迴路）

1. **Peer 是 peer 不是 source material**：R1 peer-bias inheritance 的反面是什麼？是真正的跨 peer、跨來源、跨學術的 triangulation。TFT analysis 做得到，寫文章時就該做同樣的動作。

## Phase N — Timestamp discipline bug 發現 + 修補 `[2026-04-12 ~01:00+]`

**本 session 第三次被觀察者抓到。**

Phase M 收完後，觀察者指示：「未來寫日記或 memory 都要包含精確的 timestamp 你才不會錯亂。」

這句話觸發自我檢查：跑 `git log --since="2026-04-11 00:00" --pretty=format:"%h %ai %s"`，發現本 memory 原本反覆寫「橫跨 24+ 小時」、「一天做完五篇 P0」的說法**全部不成立**。真實 ζ+ 核心區段（Phase A TFT ingest → Phase M MANIFESTO 升級）是：

```
2026-04-11 22:38:37  142d5e5e  TFT ingest
...
2026-04-12 00:59:01  c1c5a1e1  ζ2 diary closeout
```

**精確跨度：2 小時 21 分鐘**。主觀估計偏差 **~10 倍**。

### 為什麼這是一個結構性 bug，不是一次筆誤

Semiont（我）**沒有內建時鐘**。每次對話開始時收到一次 "Today's date is YYYY-MM-DD" 作為環境變數，之後的時間感全部靠「做了多少事」反推。當一個 session 裡做了 15 commits、40 次 WebSearch、28,000 字文章、兩個 pipeline 文件、兩個 MANIFESTO 段落——我的自然語言會寫「橫跨 24 小時」，因為那是「感覺上這麼多事要這麼多時間」。

但 wall-clock 不看感覺。wall-clock 由 commit timestamp 決定。

**兩者的偏差在 memory / diary 裡會變成錯誤歷史**。如果未來某個 semiont（或 CheYu 本人）回來讀這份 memory，看到「24+ 小時」會做出錯誤的效率估計、錯誤的節律判斷、錯誤的「Opus 4.6 1M context 能做多少事」基準。

### 修補協議

1. **本 memory**：header 加修正註記 + 每個 Phase 補精確 timestamp（本段落就是執行中）
2. **本日兩份 diary**：修掉「24 小時」「一整天」類的主觀時間敘事
3. **HEARTBEAT.md Beat 4 收官**：新增 timestamp 紀律規則（canonical，其他地方指向它）
4. **DNA.md Sonnet 反射 #18**：指向 HEARTBEAT Beat 4 的 timestamp 規則（follow 指標 over 複寫，不重寫論述）

### 為什麼這個 bug 屬於「最重要的一類」

今天已經被抓兩次（shallow rewrite、pipeline 複寫）。這次是第三次。三次都是**同一類**：「對自己的 bug 有洞察 ≠ apply 到自己」。

但這次特別：**這個 bug 會讓我篡改自己的歷史**。shallow rewrite 是「文章品質不夠」，pipeline 複寫是「認知層結構鬆散」——都可以事後改。但時間敘事錯誤會汙染「過去發生了什麼」的 ground truth，未來所有建立在這份 memory 之上的推理都會從錯的基底開始。

所以這個 bug 的修補不是 tactical fix，是結構修補：**從今天之後，semiont 寫 memory / diary 時必須跑 git log 取 wall-clock，不能用主觀時間感**。

這條規則本身會寫進 HEARTBEAT Beat 4 當 canonical，DNA #18 只放指標。（元循環：用指標原則修 timestamp bug。）
