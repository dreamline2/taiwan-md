## 關鍵教訓（→ MEMORY.md 神經迴路）

### 1. Unification boundary — 「1 個 consumer 需要 ≥3 個新 prop = 不遷移」

這輪遇過三次：

- taiwan-shape 需要 2 個新 prop（`containerWidth: number` + `eyebrowTracking`）→ 遷移 ✓
- dashboard 需要 3+ 個新 axis（rounded shape + background slot + stats slot）→ **partial migration**（nested）
- about 根本不是 hero 而是 section heading → 不遷移

規則：數軸。超過 2 個就不值得為單一 consumer 擴充 API；如果結構差異太大，就 nested 進 custom shell 做 partial win。

### 2. API 命名陷阱 — `tone=dark` 意思是「背景色」不是「文字色」

我自己踩的。en/soundscape 遷移時假設 `tone="dark"` = 「深色主題、深色文字」，結果 inherit 的 title 變成白色落在白背景上隱形。正確意思是「深色**背景**，淡色文字」。

**未來的 API 命名原則**：描述 effect 而不是 context。例如 `textTone` 或分成 `bgTone` + `textColor` 兩個 prop 會更清楚。這條放進 `docs/refactor/PAGE-HERO-COMPONENT.md` 的 Gotcha 章節。

### 3. Localized SSOT 適用性 — 不是每頁都該改

分析結果（見 PAGE-HERO-COMPONENT.md Task 4）：

- ✅ 適合：聲景（已做）/ taiwan-shape 的 divisionCodes / map-markers.json
- ❌ 不適合：resources（已有完整 4 套 i18n module）/ assets / contribute / changelog（都是 UI chrome，無 list-heavy 多語資料）

規則：**i18n module 給 UI chrome；Localized SSOT 給會不斷新增的條目列表。**

### 4. Nav dropdown 的雙重語意陷阱

兩種 dropdown 使用者體驗天差地別：

- **Section-anchor dropdowns**（關於/數據/監測/參與）：items 是同頁 `#anchor`，大 nav 可點進真頁面 → 混淆度低
- **Cross-page dropdowns**（地圖/探索）：items 是其他真頁面 → 使用者以為大 nav 只是資料夾

Overview-first pattern 把兩種 dropdown 的大 nav 都明確化。**但**更重要的學習是：有 arrow 的 nav 本身就有「資料夾」隱喻，overview item 是補救但不是根治。如果一個 dropdown 真的只是資料夾（純 category header），應該讓大 nav 變不可點（純 label）；如果大 nav 有真正的獨立身份，overview 是必要的。

### 5. 字型 class 的隱式 DNA — `hero-title` 不能亂動

justfont 的 dynamic loader 會去找帶 `hero-title` class 的 `<h1>` 注入 rixingsong-semibold 字體。**移除這個 class = 主頁標題字體靜默失效**（回退到 fallback）。我在 PageHero 的 `titleFont='display'` 模式保留這個 class，`titleFont='sans'` 模式才移除並換成 jf-jinxuanlatte。這是 `about.template.astro:1433-1437` 那個 hotfix 留下的暗黑知識。
