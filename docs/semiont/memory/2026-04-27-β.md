### 一、四 session 連續傳遞 XS task 的腐蝕

`.claude/launch.json` + `scheduled_tasks.lock` 兩條 gitignore 任務從 2026-04-26 ε commit `92d1c5b2` 誤 add → ζ memory「可順手做」→ α memory「仍 pending」→ β 才動手。**四 session 對一個 ~30 秒能完成的任務**。

問題不在 task 大小，是在「順手做」這個 framing：

- ζ 寫「可順手做」當下沒做，因為 ζ 主任務是 PR queue cleanup
- α 寫「仍 pending」當下沒做，因為 α 主任務是 7 PR triage
- β 動手是因為 morning light tick 沒有大主任務

**模式**：「**XS task 的真實成本不是 30 秒，是被 4 session 帶過的 attention overhead**」。每次 session 讀 handoff list 都掃過這條，但都判定「優先序低」。如果單獨看，這條成本是 30 秒；累積看，這條已耗 4 session × ~5 sec attention = 20 sec attention overhead，加上每次 commit 都 dirty git status `.claude/launch.json` 1 行、累積讀者眼球的 noise。

**校準候選**：handoff 裡標 XS 且不需 context 的任務，下次讀到第二次就強制處理（不能傳第三次）。如果單獨講「這次 light tick 不做 XS」是合法的，但「四次都不做 XS」就成了結構性 procrastination。對應 LESSONS-INBOX 候選教訓。
