## 本 session 新長出來的神經迴路

### 1. `httpRequestsAdaptiveGroups` 的 userAgent 維度在 Free tier 可用

**這是最大的認知翻轉**。先前的心跳（α session）我自己得出的結論是「Free tier 抓不到 AI crawler breakdown，只有 Pro/Enterprise 的 Bot Management 才行」。這個結論來自單一資料點：`botManagementVerifiedBot` 欄位報錯 `unknown arg`。

正確的切法是：**Enterprise-only 的是 `botManagement.*` 欄位群，不是整個 `httpRequestsAdaptiveGroups` 端點**。`userAgent` 維度在所有 plan 都能用。

**觸發反轉的事件**：哲宇貼了一張 Cloudflare dashboard 的 AI Crawl Control 截圖，清楚顯示 Free tier 也有完整的 crawler 分解（Amazon 1.5k / Huawei 770 / OpenAI 595 / ...）。那個截圖就是證據——「Cloudflare 官方 UI 都能顯示了，API 怎麼可能不給」。

**然後才去翻文檔**：https://developers.cloudflare.com/ai-crawl-control/reference/graphql-api/ — 明確說「userAgent 維度所有 plan 可用，要更精準的 `botDetectionIds` 才需要 Bot Management」。

**神經迴路教訓**：

- **單一 API error 不是結論，是假設**。`botManagementVerifiedBot` 報錯 ≠ 所有 user-agent 分解都壞。應該退一步測其他欄位。
- **使用者的 UI 截圖 = 產品真實 capability 的證據**。如果使用者能在 Cloudflare dashboard 看到這個數字，那 API 一定有某個方式能拿到——不然 dashboard 本身怎麼 render。
- **文檔 before 實驗 before 結論**。我先前的順序是「實驗 → 結論 → 沒讀文檔」。應該是「文檔 → 實驗 → 結論」。這個順序倒轉讓我多花了一週才解鎖這個 endpoint。

### 2. Free tier quota limit: `httpRequestsAdaptiveGroups` 限 1 日 window

實作之後馬上撞到 `cannot request a time range wider than 1d`。這是 Cloudflare 的 Free tier quota（Pro 應該是 7d，Enterprise 30d+）。

**解法**: 迴圈 7 次，每次查 1 日，客戶端 aggregate：

```python
for offset in range(days):
    day_end = now - timedelta(days=offset)
    day_start = day_end - timedelta(days=1)
    ...
```

**加上 soft-fail 機制**: 某一天 query 失敗（例如 transient network error）不能 kill 整個 fetch。新增 `_cf_graphql_soft()` wrapper，回傳 `(data, err)` tuple 而不是 `fail()` exit。追蹤 `daysFetched` / `daysFailed` 給 diagnostic 用。

**神經迴路教訓**：凡是「免費計畫的 limit」都要假設會遇到。Pre-commit 寫 Cloudflare fetcher 時 就該把 quota 假設列入 design，不要「先跑跑看」撞到才想辦法。

### 3. `userAgent_like: "%bot%"` 伺服端過濾會漏掉 crawler

第一版我用 `userAgent_like: "%bot%"` 當伺服端 pre-filter 想省 10k row limit，結果測出來 total=9,572。移除這個過濾之後，total=11,508（多 ~2,000）。

**原因**: SQL `LIKE %bot%` 是大小寫敏感的（至少在 Cloudflare 的實作上）。沒有 "bot" 字串的 crawler 就被漏掉——像 `FacebookBot`（其實 UA 是 `facebookexternalhit`）、`Bytespider`、`Perplexity-User`。

**正確做法**: 不做伺服端 pre-filter，**全部抓回客戶端再用 regex pattern list 精準 match**。對 taiwan.md 這種規模（~30k req/day，~200-500 unique UAs），遠低於 10k 行上限，完全抓得完。

**神經迴路教訓**：伺服端過濾的「精準度 vs 成本」權衡裡，對小網站來說精準度幾乎永遠贏。只有當資料真的大到 client-side 跑不動才該開始犧牲精準度。

### 4. D3 word cloud — 在 Astro 裡載外部 JS 的正確姿勢

`<script is:inline defer src="https://cdn.jsdelivr.net/npm/d3@7.9.0/dist/d3.min.js"></script>` — 兩個關鍵字：

- **`is:inline`**: 告訴 Astro 不要把這個 script 當 ES module 打包。直接輸出 `<script src=...>`，走瀏覽器原生 script loading。
- **`defer`**: 不 block first paint，但保證在 DOMContentLoaded 之前完成。對 `waitForD3()` polling 友善。

然後在 `<script define:vars={{lang}}>` 那個主 render script 裡用 `waitForD3()` Promise（50ms polling interval, 5s timeout）等 `window.d3 && window.d3.layout.cloud` 同時存在再呼叫。這樣不論 d3 的兩個 script 誰先載完都能安全開始 layout。

**Seeded random 的必要性**: d3.layout.cloud 用 `Math.random()` 決定 archimedean spiral 的起點和 rotation。每次 reload 就重新 random → 文字雲位置跳來跳去，視覺體驗很糟。加一個 deterministic PRNG（`seed = (seed*9301 + 49297) % 233280`）並傳給 `.random(seededRandom)` 就 lock 住了。

**對 CJK 的字體選擇**: `"Noto Sans TC", "Source Han Sans TC", system-ui` — 用 Noto Sans TC 而不是 serif，中文字在文字雲裡 serif 會太複雜難讀。

## 本 session 累積的 Sonnet 反射

加在 DNA v1.1 「Sonnet 特別留意」八條的延伸（還沒寫進 DNA.md 但先在這裡記錄）：

**9. 長任務先開 worktree**：如果預期會 touch 多個目錄、會跑 build、會超過 30 分鐘—第一個動作就是 `git worktree add`。`11ad6bed` 是忽略這條的代價。

**10. API error ≠ capability 界線**：單一欄位報錯不等於整個 endpoint 不可用。先退一步測其他欄位，再決定要不要放棄整個 API。Cloudflare `botManagementVerifiedBot` 報錯 → 我錯誤推斷「整個 bot 分解不可用」→ 花了兩週才發現 `userAgent` 維度一直都在。

**11. 使用者的 UI 截圖 = 產品真實 capability 證據**：如果使用者能在 vendor 的 dashboard 看到某個數字，那個數字一定有某種 API 路徑能拿到。不然那個 UI 本身怎麼 render。把截圖當作「已證明存在」的證據，而不是「看起來很像我要的東西」。

**12. 「再小一點」是 log scale 的訊號**：字體大小、地圖 marker 大小、bar chart —— 當使用者說「熱門的要更突出、冷門的要更小」，他在描述 log scale，不是線性 scale。直接上 log，而且不要用 min floor 把最小值墊高（`log(x+1)` 會讓最小值不等於 0）。用 `log(max(x, 1))` 讓 x=1 真的落在 0。
