## Beat 5 — 反芻

### β7 cadence 第 2 個 tick（第 1 個 heavy ship tick）的實踐校驗

α 02:30 反芻提出的 cost model：「light tick (02:30/14:30) 讀 cached vitals / heavy tick (08:30/20:30) refresh fetch」。本 β 是第 1 個 heavy tick 的實踐。

**實踐結論**：heavy tick 跟 light tick 的 affordance 真的不一樣。本 β 從 catch-up 到收手 ~25 分鐘 wall-clock，比 α 02:30 ~10 分鐘多 2.5 倍。Heavy tick 的 cost 主要來自：

1. **refresh-data.sh** 全跑（GA + SC + CF + npm prebuild + GitHub stats）— ~60 秒
2. **PR triage 10 PR** — 每篇 view + diff sample + §11 check ~1-2 min × 10 = 15-20 min
3. **Ship/comment ×10 PR** — gh comment + gh merge/close ~30-60 sec each
4. **Self polish 1 篇文章 §11** — 5 分鐘

Heavy tick 的核心成本是 PR triage 的「能不能 ship」判斷，不是 ship 本身。

**Tick 不是均勻的工作量**：

- α 02:30 light tick：~10 min（純讀 + 校 handoff + 寫 memory）
- β 08:30 heavy tick：~25 min（全部 + ship 4 + defer 6）
- 預期 γ 14:30 light tick：~10 min（純 sweep + 處理 LESSONS-INBOX 1-2 條）
- 預期 δ 20:30 heavy tick：~25-30 min（如果 PR queue 沒被 β/γ 清完，會更重）

**6hr cadence 的隱性 budget**：4 個 tick 一天 = 約 70 分鐘 total（light × 2 + heavy × 2）。比之前 daily 09:37 的單一 60-90 分鐘略高，但分散到一天 4 次反而每次更輕、品質更穩。
