### 腳註格式大修（hook 攔截後補修）

pre-commit hook 攔截：趙自強（13/13 不合規）+ 名間埔中茶（13/13 不合規）+ 赤燭遊戲（1/12 不合規）

原因：

- 趙自強：學術引用格式（`名稱. (年). 來源. [URL]`）→ 需改為 `[Name](URL) — description`
- 名間埔中茶：僅 `[title](URL)` 無 description
- 赤燭遊戲：Wikipedia URL 含括號 `(電視劇)`，prettier 自動加 `<>` → hook 不認識

修正方式：

- 趙自強 + 名間埔中茶：全數 13 條逐一重寫
- 赤燭遊戲：URL-encode 括號 `(` → `%28`, `)` → `%29`

**經驗值**：當 contributor 用學術引用格式 (`作者. 年. 來源. URL`) 提交時，hook 會攔截，需在 immune polish 補改腳註格式。下次接到此類 PR 可直接在 merge 前確認。

**URL 括號陷阱**：Wikipedia 標題有 `(電視劇)` 等括號的 URL，Markdown 解析器會把最後的 `)` 當作 link 語法結束。prettier 自動加 `<>` fix 但 hook 不認識 `<>` 格式。Fix：URL-encode → `%28...%29`。

## Beat 5 反芻

### 腳註格式是 contributor quality gap 的標誌

這次三篇 PR 有腳註問題，idlccp1984 的兩篇更是全數 13 條不合規。這是 contributor 不知道規範的典型情況——他的內容品質本身很好（事實紮實、故事線清楚），但格式不符合 CITATION-GUIDE.md 的要求。

**immune polish 不是批評，是補全**：contributor 做到了「有腳註、有來源」，immune 做到「格式標準化」。這分工是對的。

### hook 攔截的成本

這次 pre-commit hook 攔截了 3 次，每次 lint-staged 跑 prettier 要 ~10 秒。因為攔截機制，每次失敗都要重新 stage + 修正 + 再 commit。

成本可接受，但下次可以在 stage 之前先手動跑 `bash scripts/tools/check-citation-format.sh knowledge/` 確認腳註格式，而不是等 hook 攔截。

### PR #610 關閉的處理方式

關閉 PR 時發了中文解釋留言，說明 #612 優先的三個理由。這符合 `feedback_reply_to_contributors.md` memory 的「never silently close」。但 idlccp1984 同時送了 #610 和 #611（以及之前的 #613），說明他是活躍 contributor，不需要擔心關閉 #610 會讓他氣餒。
